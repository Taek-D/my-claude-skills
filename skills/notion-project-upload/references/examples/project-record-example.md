# 프로젝트 원장 예시: 쿠팡윙 가격 모니터링 자동화

> 이 파일은 "직군 색 없이 팩트만 최대한 상세하게" 기록하는 원장 작성 예시다.
> DA/PA/BA/PM 어떤 포트폴리오를 만들어도 여기서 꺼내 쓸 수 있어야 한다.

---

## 🎯 Performance Overview

| Metric | Before | After | 비고 |
|--------|--------|-------|------|
| 가격 확인 소요 시간 | 40분/일 | 0분 (자동) | 6개 플랫폼 기준 |
| 모니터링 범위 | 2-3개 플랫폼 | 6개 플랫폼 | 무신사/29CM/쿠팡/올리브영/SSG/롯데온 |
| 최저가 포착 타이밍 | 수시간~1일 지연 | 5분 이내 | 스케줄러 5분 주기 |
| 수동 오류율 | 약 12% | 0.3% | 스프레드시트 수기 입력 대비 |
| 월 운영 비용 | 0원 (수작업) | ₩3,200 | Supabase Pro 플랜 |

---

## 🔍 Problem

**배경**

쿠팡윙 드랍쇼핑 운영 중. 동일 상품을 무신사·쿠팡·올리브영 등 6개 플랫폼에 올려놓고
타 플랫폼 최저가보다 낮게 유지해야 쿠팡 알고리즘 노출에 유리함.

**문제**

매일 아침 6개 플랫폼을 직접 접속해 가격을 확인하고 구글 스프레드시트에 기록하는 작업이
40분 이상 소요됨. 이 작업이 매일 반복되고, 낮 시간대 가격 변동은 아예 감지하지 못하고 있었음.
수기 입력 오류로 인한 잘못된 가격 기준 설정이 월 2-3회 발생.

**왜 문제가 심각했는가**

상품 수가 늘수록 확인 시간이 선형으로 증가하는 구조. 당시 상품 40개 기준 40분이었고
100개로 늘어나면 100분+ 소요 예상. 스케일이 안 되는 구조였음.

---

## 🛠️ Approach

**검토한 대안**

| 방법 | 검토 이유 | 기각 이유 |
|------|----------|----------|
| 노코드 자동화 (Zapier/Make) | 빠른 구축 | 크롤링 불가, API 없는 플랫폼 대응 안 됨 |
| 외부 가격 비교 서비스 구독 | 구축 불필요 | 드랍쇼핑 특화 서비스 없음, 월 ₩50K+ 비용 |
| Google Sheets + Apps Script | 익숙한 환경 | JS 크롤링 한계, 반정형 페이지 파싱 불안정 |
| **Python + Playwright + Supabase** | 유연한 크롤링 가능 | — | ✅ 선택 |

**선택 근거**

Playwright는 SPA 렌더링까지 처리 가능해 JS 렌더링 기반 쇼핑몰(무신사 등)에 대응 가능.
Supabase는 무료 티어로 시작 가능하고 PostgreSQL 기반이라 이후 분석 쿼리 작성이 용이.
Python은 기존 자동화 경험 있어 러닝커브 없음.

---

## ⚙️ Solution

**시스템 구성**

```
스케줄러 (APScheduler, 5분 주기)
    ↓
플랫폼별 크롤러 (Playwright, 어댑터 패턴)
    ↓
가격 파싱 → Supabase 저장
    ↓
가격 변동 감지 → Discord Webhook 알림
    ↓
Google Sheets 동기화 (기존 운영 방식과 병행)
```

**핵심 구현 내용**

- 플랫폼별 크롤러를 어댑터 패턴으로 분리 → 신규 플랫폼 추가 시 어댑터 1개만 작성하면 됨
- Playwright headless 모드, `wait_until="networkidle"` 전략으로 SPA 렌더링 대기
- 가격 변동 임계값 설정 가능 (기본: ±3% 이상 변동 시 알림)
- GitHub Actions cron으로 서버리스 운영 (로컬 서버 불필요)
- Supabase RLS 설정으로 데이터 보안

**코드 구조**

```
monitors/
├── base.py          # BaseMonitor 추상 클래스
├── musinsa.py       # 무신사 어댑터
├── coupang.py       # 쿠팡 어댑터 (HMAC 인증 포함)
├── oliveyoung.py    # 올리브영 어댑터
└── ...
scheduler.py         # APScheduler 진입점
notifier.py          # Discord Webhook
sync.py              # Google Sheets 동기화
```

**운영 현황**

- 일 평균 크롤링 횟수: 288회 (5분 × 6플랫폼 × 8시간 운영)
- Supabase 저장 레코드: 약 8,600건/월
- GitHub Actions 사용량: 월 약 1,400분 (무료 티어 2,000분 이내)

## 🔧 Challenge

**Challenge 1: 쿠팡 HMAC 인증**

쿠팡은 단순 크롤링이 불가하고 파트너 API를 써야 함. HMAC-SHA256 서명 방식이 처음이라
서명 생성 로직 디버깅에 3일 소요. 타임스탬프 포맷 오류(밀리초 vs 초)가 원인이었음.
공식 문서의 예제 코드가 Python 2 기준이었던 것도 혼란 요소.

**Challenge 2: 무신사 DOM 구조 변경**

크롤러 배포 3주 후 무신사가 가격 영역 클래스명을 변경 → 파싱 실패.
오류 감지까지 6시간이 걸렸음 (알림 없었기 때문).
→ 이후 가격 파싱 실패 시 Discord 에러 알림 추가, 셀렉터 후보 복수 지정으로 대응.

**Challenge 3: GitHub Actions 타임아웃**

Playwright 크롤링이 느린 플랫폼(SSG)에서 간헐적 타임아웃 발생.
Actions 기본 타임아웃 6분으로는 부족한 경우가 있었음.
→ 플랫폼별 타임아웃 개별 설정, 재시도 로직(최대 3회 + exponential backoff) 추가.

**피벗 기록**

초기에는 가격을 Google Sheets에만 저장하는 구조로 시작했으나,
Sheets API 할당량(분당 60회)이 5분 주기 크롤링과 충돌함.
→ Supabase를 primary storage로 변경, Sheets는 일 1회 동기화로 전환.

## 📊 Result

**정량 성과**

- 일 40분 수작업 완전 제거 → **월 20시간 절약**
- 수기 입력 오류율 12% → **0.3%** (파싱 실패 알림으로 즉시 감지)
- 가격 변동 감지 타이밍: 수시간~1일 → **5분 이내**
- 모니터링 플랫폼 수: 3개 → **6개** (동일 공수)
- 월 운영 비용: **₩3,200** (Supabase Pro)

**정성 성과**

- 상품 수 40개 → 120개로 늘려도 운영 공수 동일 (스케일 문제 해결)
- 가격 변동 히스토리 데이터 6개월치 축적 → 플랫폼별 할인 패턴 분석 가능해짐
- 크롤링 실패 즉시 알림으로 데이터 공백 기간 거의 없음

**아직 해결 안 된 것**

- 롯데온 크롤러 불안정 (월 2-3회 파싱 실패) → 셀렉터 재점검 필요
- 가격 히스토리 기반 "최저가 예측" 기능 미구현

## 💡 Learning

**기술적으로 배운 것**

어댑터 패턴의 실용성을 직접 체감. 처음부터 추상 클래스를 만들어둔 덕분에
3번째 플랫폼부터는 어댑터 작성에 2-3시간이면 됐음.
반대로 HMAC 인증처럼 "공식 문서만 봐서는 안 되는" 케이스가 있다는 것도 배움
— 실제 동작하는 레퍼런스 코드를 먼저 찾는 게 효율적.

**운영적으로 배운 것**

에러 알림 없이 운영하면 장애를 한참 뒤에 발견한다.
크롤링 성공/실패 여부를 Discord로 리포트하는 것만으로
문제 감지 시간이 6시간 → 5분으로 줄었음.

**다음에 적용할 것**

셀렉터 변경에 취약한 구조를 LLM 기반 동적 파싱으로 개선하는 것을 검토 중.
페이지 HTML을 Claude API에 넘겨 가격 영역을 추출하는 방식 — DOM 구조 변경에 강인.
